# Sentinel Path: технический разбор

Этот документ объясняет, как устроен проект изнутри и почему мы приняли именно такие решения.

## Почему архитектура разбита на модули

Мы выбрали разбиение на независимые слои, чтобы можно было:
- вызывать весь пайплайн одной командой через `SentinelEngine`;
- тестировать и переиспользовать части отдельно (`topology`, `fragility`, `simulation`);
- менять реализацию в одном слое без каскадных правок в остальных.

Это решает проблему "монолитного скрипта", где любой апдейт в расчете сразу ломает CLI, формат отчета и визуализацию.

## Компоненты

| Файл | Зона ответственности | Почему это важно |
| :--- | :--- | :--- |
| `sentinel_path/__main__.py` | CLI: принимает пути к входным файлам и запускает анализ | Дает воспроизводимый запуск из терминала без написания кода |
| `engine.py` | Фасад-оркестратор пайплайна `Topology -> Fragility -> Simulation` | Единая точка входа упрощает интеграцию и тестирование |
| `core/topology.py` | Построение DAG, CPM (ES/EF/LS/LF/TF), валидация графа | Без корректной топологии все метрики риска теряют смысл |
| `core/fragility.py` | Поиск узлов сходимости путей и расчет PCI | Показывает структурные риски, которые не видны в голом CPM |
| `stochastic/simulation.py` | Монте-Карло на Beta-PERT, cruciality, sensitivity | Переводит план из "жестких дат" в вероятностную модель |
| `models/schemas.py` | Контракты и валидация входа/выхода через Pydantic | Ловит ошибки формата до запуска тяжелых вычислений |
| `sentinel_path/path_handler.py` | Нормализация путей и проверка доступа | Снижает количество OS-специфичных падений на старте |

## Поток данных

1. CLI получает пути к `tasks/dependencies/config`.
2. Пути нормализуются и проверяются на доступность.
3. `engine.py` валидирует вход в модели Pydantic.
4. `core/topology.py` строит DAG и считает базовый CPM.
5. `core/fragility.py` ищет точки хрупкости.
6. `stochastic/simulation.py` считает вероятностные метрики.
7. `SentinelReport` возвращается в JSON; при необходимости генерируются графики.

## Нюансы и ограничения

- **Слеши в путях (Windows/Linux):** мы приводим строковые пути к единому формату, но важно учитывать, что экзотические пути (например, сетевые UNC) требуют отдельной проверки в вашем окружении.
- **Пустые пути:** если в CLI передать пустую строку, она превратится в текущую директорию через `Path.resolve()`. Это может быть неожиданно, поэтому лучше валидировать такие кейсы на уровне вызывающего кода.
- **Права доступа:** `os.access` не всегда идеально отражает поведение сетевых файловых систем; при корпоративных шарах лучше делать отдельный smoke-test.
- **Пустой граф / изолированные узлы:** обработка жесткая и валится с ошибкой — так проще поймать плохие данные раньше.
- **Детерминизм симуляции:** для стабильных результатов в CI и автотестах используйте фиксированный `rng_seed`.

## Запуск

```bash
python -m sentinel_path --tasks tasks.json --dependencies deps.json --config config.json --output report.json --charts-dir charts
```

## Типовые ошибки

| Ошибка | Что это значит | Что делать |
| :--- | :--- | :--- |
| `CycleError` | В зависимостях есть цикл | Проверить `deps.json` и убрать кольцевые связи |
| `ValidationError` | Вход не совпадает со схемами | Сверить поля и типы с `models/schemas.py` |
| `GraphTopologyError` | Пустой или несвязный граф | Проверить, что задачи действительно связаны |
| `Path is not accessible` | Нет доступа к файлу/каталогу | Проверить путь и права ОС |
